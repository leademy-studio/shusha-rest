# Регламент Frontend-разработки (HTML и CSS)

Этот документ определяет основные правила и стандарты для написания HTML и CSS кода. Соблюдение этих правил обязательно для всех frontend-программистов с целью обеспечения консистентности, читаемости и поддерживаемости кодовой базы.

## Часть 1: Регламент верстки HTML-разметки

### 1.8. Дополнительные рекомендации и best practices

#### Доступность (a11y)
- Используйте aria-атрибуты (`aria-label`, `aria-labelledby`, `aria-describedby`) для улучшения доступности.
- Применяйте `tabindex` для управления порядком фокуса.
- Используйте роли (`role="button"`, `role="dialog"` и др.) для кастомных интерактивных элементов.

#### SVG-графика
- Для иконок и сложных иллюстраций используйте inline SVG, чтобы можно было управлять стилями и доступностью.
- Добавляйте подписи и описания через `<title>` и `<desc>` внутри SVG.
- Для простых изображений используйте `<img src="...">`.

#### Ссылки
- Для внешних ссылок с `target="_blank"` всегда добавляйте `rel="noopener noreferrer"`.
- Не используйте пустые ссылки (`href="#"`) без обработчика.

#### Lazy loading
- Используйте `loading="lazy"` для изображений и `<iframe>`, чтобы ускорить загрузку страниц.

#### Формы
- Используйте атрибуты `autocomplete`, `aria-required`, `aria-invalid` для улучшения UX и доступности.
- Всегда связывайте поля с `<label>`.
- Для ошибок и подсказок используйте aria-live или визуальные индикаторы.

#### Структурированные данные (microdata, schema.org)
- Размечайте статьи, хлебные крошки, товары с помощью microdata или JSON-LD по стандарту schema.org для улучшения SEO.

#### Favicon и meta-теги
- Добавляйте favicon разных размеров и форматов.
- Используйте Open Graph и Twitter Cards для корректного отображения ссылок в соцсетях.

#### Организация JS-кода (если JS входит в проект)
- Соблюдайте структуру файлов и модульность.
- Используйте современные стандарты ES6+.
- Следуйте best practices по именованию и организации кода.

#### Тестирование верстки
- Используйте Pixel Perfect, автоматизированные тесты (например, Playwright, Cypress) для проверки качества верстки.

#### Чек-лист для ревью верстки
- Перед сдачей задачи проходите чек-лист: семантика, доступность, адаптивность, кроссбраузерность, валидность, производительность.

#### Типовые ошибки и их решения
- Приведите примеры частых ошибок (например, отсутствие alt у изображений, неправильная иерархия заголовков) и способы их исправления.
### 1.7. Семантические теги для специфических элементов

Используйте специальные HTML5-теги для элементов, которые имеют определённое семантическое значение и не должны размечаться обычным `<div>`.

- **Хлебные крошки (Breadcrumbs):**
    - Используйте тег `<nav>` с атрибутом `aria-label="breadcrumb"` для блока хлебных крошек.
    - Внутри используйте ненумерованный список `<ul>` или список `<ol>`, где каждый элемент — это `<li>` с ссылкой `<a>` (кроме последнего).
    - Пример:
    ```html
    <nav aria-label="breadcrumb">
        <ol class="breadcrumb">
            <li class="breadcrumb__item"><a href="/">Главная</a></li>
            <li class="breadcrumb__item"><a href="/blog/">Блог</a></li>
            <li class="breadcrumb__item breadcrumb__item--active" aria-current="page">Публикация</li>
        </ol>
    </nav>
    ```

- **Публикации в блоге, новости, отдельные карточки статей:**
    - Используйте тег `<article>` для каждого отдельного материала (пост, новость, карточка товара и т.д.).
    - Внутри `<article>` используйте заголовки (`<h2>`, `<h3>`), метаинформацию (`<time>`, `<address>`, `<span class="author">`), основной текст и т.д.
    - Пример:
    ```html
    <article class="blog-post">
        <h2 class="blog-post__title">Заголовок публикации</h2>
        <div class="blog-post__meta">
            <span class="author">Автор: Иван Иванов</span>
            <time datetime="2025-12-20">20 декабря 2025</time>
        </div>
        <div class="blog-post__content">
            <p>Текст публикации...</p>
        </div>
    </article>
    ```

- **Навигация:**
    - Для любых блоков с навигационными ссылками используйте тег `<nav>`, а не `<div>`.

- **Списки:**
    - Для перечислений используйте `<ul>`, `<ol>`, `<dl>` вместо `<div>`.

- **Формы:**
    - Для форм используйте тег `<form>`, а не `<div>`.

- **Таблицы:**
    - Для табличных данных используйте тег `<table>` и связанные с ним (`<thead>`, `<tbody>`, `<tfoot>`, `<tr>`, `<td>`, `<th>`), а не `<div>`.

- **Медиа-контент:**
    - Для вставки изображений используйте `<img>`, для видео — `<video>`, для аудио — `<audio>`, для иллюстраций — `<figure>` и `<figcaption>`.

**Общее правило:** Если для блока существует специальный семантический тег — используйте его вместо `<div>`. Это улучшает доступность, SEO и структуру документа.
### 1.1. Общая структура страницы

Каждый HTML-документ должен следовать стандартной структуре HTML5.

```html
<!DOCTYPE html>
<html lang="ru">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Название страницы</title>
    <link rel="stylesheet" href="css/style.css">
</head>
<body>
    <!-- Содержимое страницы -->
</body>
</html>
```

-   `<!DOCTYPE html>`: Обязательное объявление типа документа.
-   `<html lang="ru">`: Указывайте основной язык содержимого страницы для улучшения доступности и SEO.
-   `<meta charset="UTF-8">`: Обязательная кодировка для корректного отображения символов.
-   `<meta name="viewport" ...>`: Обязательный тег для управления областью просмотра на мобильных устройствах и реализации адаптивного дизайна.

### 1.2. Семантическая разметка и структура секций

#### 1.2.1. Использование семантических тегов

Каждый логический блок на странице должен быть обернут в соответствующий семантический тег. Это улучшает структуру документа, доступность (accessibility) для скринридеров и SEO.

-   `<header>`: "Шапка" сайта или секции. Обычно содержит логотип, навигацию, контакты.
-   `<main>`: Основное, уникальное содержимое страницы. Должен быть только один тег `<main>` на странице.
-   `<footer>`: "Подвал" сайта или секции. Содержит копирайт, ссылки, контактную информацию.
-   `<nav>`: Блок с основной навигацией по сайту.
-   `<section>`: Крупный смысловой раздел документа. Каждая секция, как правило, должна иметь свой заголовок (`<h2>` - `<h6>`).
-   `<article>`: Автономный, независимый фрагмент контента (например, статья в блоге, новость, карточка товара).
-   `<aside>`: Дополнительное содержимое, которое не является основной частью контента (например, боковая панель, сноски, рекламные блоки).

#### 1.2.2. Контейнер для ограничения ширины

Для ограничения ширины контента и его центрирования используется специальный класс-контейнер.

-   **Название класса:** `.container`
-   **Назначение:** Задает максимальную ширину (`max-width`) для рабочей области и центрирует ее с помощью `margin: 0 auto`.

Пример CSS для контейнера:
```css
.container {
    max-width: 1200px; /* или другая ширина по макету */
    margin-left: auto;
    margin-right: auto;
    padding-left: 15px;  /* Боковые отступы для безопасности на малых экранах */
    padding-right: 15px;
}
```

#### 1.2.3. Расположение контейнера

**Ключевое правило:** Контейнер (`<div class="container">`) всегда должен находиться **внутри** тега секции (`<section>`, `<header>`, `<footer>` и т.д.).

Это позволяет стилизовать фон секции (цвет, градиент, фоновое изображение) на всю ширину экрана, в то время как содержимое внутри контейнера останется аккуратно сгруппированным и ограниченным по ширине.

✅ **Правильно:**

```html
<!-- Фон секции .promo может быть растянут на 100% ширины экрана -->
<section class="promo">
    <div class="container">
        <h1>Промо-заголовок</h1>
        <p>Текст внутри контейнера с ограниченной шириной.</p>
    </div>
</section>
```

❌ **Неправильно:**

```html
<!-- Фон секции .promo будет ограничен шириной контейнера, что неверно -->
<div class="container">
    <section class="promo">
        <h1>Промо-заголовок</h1>
        <p>Текст.</p>
    </section>
</div>
```

### 1.3. Именование CSS-классов

Рекомендуется использовать методологию **БЭМ (Блок, Элемент, Модификатор)** для именования классов. Это делает CSS-код структурированным, предсказуемым и легко масштабируемым.

-   **Блок:** Независимый компонент (`.card`, `.menu`, `.form`).
-   **Элемент:** Часть блока, которая не может существовать отдельно от него (`.card__title`, `.menu__item`). Имя элемента отделяется от имени блока двумя подчеркиваниями (`__`).
-   **Модификатор:** Вариация блока или элемента, определяющая его внешний вид или состояние (`.card--dark`, `.menu__item--active`). Имя модификатора отделяется двумя дефисами (`--`).

**Пример:**

```html
<form class="search-form search-form--header">
    <input class="search-form__input" type="text" placeholder="Поиск...">
    <button class="search-form__button search-form__button--disabled" type="submit">Найти</button>
</form>
```

### 1.4. Заголовки

-   На странице должен быть только **один** заголовок `<h1>`. Обычно это название страницы или главный заголовок на первом экране.
-   Заголовки должны использоваться в строгой иерархии (`h1` -> `h2` -> `h3`...). Нельзя перескакивать уровни (например, после `h2` сразу использовать `h4`).
-   Каждая `<section>` должна начинаться с заголовка своего уровня (чаще всего `<h2>`).

### 1.5. Доступность (Accessibility, a11y)

Веб-доступность (также известная как a11y) — это практика создания веб-сайтов, которыми могут пользоваться все люди, независимо от их физических или технических ограничений.

#### 1.5.1. Изображения

-   **Атрибут `alt`:** Все теги `<img>` должны иметь атрибут `alt`.
    -   Если изображение несет смысловую нагрузку (например, фотография товара, диаграмма), в `alt` должно быть краткое, но точное описание того, что изображено.
    -   Если изображение является чисто декоративным (например, фоновый узор, иконка, дублирующая текст), атрибут `alt` должен быть пустым (`alt=""`). Это укажет скринридерам проигнорировать изображение.

    ✅ **Правильно (смысловое изображение):**
    ```html
    <img src="chart.png" alt="График роста продаж за последний квартал.">
    ```

    ✅ **Правильно (декоративное изображение):**
    ```html
    <img src="background-pattern.svg" alt="">
    ```

#### 1.5.2. Интерактивные элементы (ссылки, кнопки, поля ввода)

-   **Семантика:** Используйте правильные теги для интерактивных элементов.
    -   `<a>`: для навигации (переход на другую страницу или якорь на текущей).
    -   `<button>`: для выполнения действий на текущей странице (отправка формы, открытие модального окна, запуск/остановка медиа).
-   **Доступность с клавиатуры:** Все интерактивные элементы должны быть доступны и управляемы с помощью клавиатуры. Убедитесь, что пользователь может перемещаться по всем ссылкам, кнопкам и полям с помощью клавиши `Tab`.
-   **Видимый фокус:** Никогда не убирайте `outline` для состояний `:focus` без предоставления альтернативного, хорошо заметного стиля фокуса. Пользователи, которые используют клавиатуру для навигации, должны видеть, какой элемент активен в данный момент.

    ❌ **Неправильно:**
    ```css
    :focus {
        outline: none;
    }
    ```

    ✅ **Правильно (кастомный стиль фокуса):**
    ```css
    .button:focus {
        outline: none;
        box-shadow: 0 0 0 3px rgba(0, 123, 255, 0.5);
    }
    ```
-   **Связь с метками:** Все элементы форм (`<input>`, `<textarea>`, `<select>`) должны быть связаны со своими текстовыми метками с помощью тега `<label>` и атрибута `for`.

    ✅ **Правильно:**
    ```html
    <label for="user-name">Ваше имя:</label>
    <input type="text" id="user-name" name="name">
    ```

### 1.6. Шаблонизация компонентов

Для повышения переиспользуемости и упрощения поддержки кода рекомендуется подходить к верстке с точки зрения компонентов. Компонент — это изолированный, самодостаточный блок интерфейса, который можно использовать в разных частях проекта.

#### 1.6.1. Принципы компонентного подхода

-   **Независимость:** Компонент не должен зависеть от своего окружения. Вся необходимая информация (текст, ссылки, изображения) должна передаваться ему извне.
-   **Переиспользуемость:** Один и тот же компонент (например, кнопка, карточка, поле ввода) должен без изменений работать в разных секциях сайта.
-   **Изоляция:** Стили и логика компонента не должны влиять на другие элементы страницы. Методология БЭМ отлично помогает в достижении этой цели.

#### 1.6.2. Пример шаблона компонента

Рассмотрим компонент `card` из примера ниже. Его можно вынести в отдельный файл (в случае использования шаблонизаторов типа Pug, Nunjucks или фреймворков) или просто иметь его структуру как «шаблон» для копирования.

**Шаблон компонента `card`:**

```html
<!-- 
  Компонент: card
  Модификаторы:
    - card--dark: темная тема
  Принимает:
    - title (string): Заголовок карточки
    - text (string): Текст карточки
-->
<div class="card">
    <h3 class="card__title">{title}</h3>
    <p class="card__text">{text}</p>
</div>
```

Такой подход позволяет:
-   Быстро создавать новые элементы интерфейса на основе готовых блоков.
-   Легко вносить изменения: достаточно поправить один шаблон, и изменения применятся везде, где он используется.

## Часть 2: Регламент оформления CSS-файлов

Этот раздел дополняет регламент по HTML и определяет стандарты написания CSS-кода.

### 2.1. Структура файлов и стратегия загрузки

Для удобства разработки и поддержки рекомендуется разделять CSS-код на несколько логических файлов. Это помогает содержать код в порядке и не требует использования сложных инструментов.

#### 2.1.1. Структура для разработки (Разбивка на файлы)

Разделяйте стили на небольшие, сфокусированные файлы по их назначению.

Пример структуры:
-   `normalize.css`: Сброс или нормализация стилей браузера.
-   `main.css` или `base.css`: Основные стили проекта (переменные CSS Custom Properties, стили для `body`, `a`, заголовков и т.д.).
-   `layout.css`: Стили для основных блоков макета (контейнер, шапка, подвал).
-   `components.css`: Стили для всех переиспользуемых компонентов (кнопки, карточки, формы).

Эти файлы подключаются в HTML-документе в нужном порядке:

```html
<head>
    ...
    <link rel="stylesheet" href="css/normalize.css">
    <link rel="stylesheet" href="css/main.css">
    <link rel="stylesheet" href="css/layout.css">
    <link rel="stylesheet" href="css/components.css">
    ...
</head>
```

**Важное замечание:** Хотя такая разбивка удобна для разработки, для рабочего сайта множество CSS-файлов может замедлить загрузку из-за большого количества HTTP-запросов. Поэтому перед публикацией сайта эти файлы следует объединить (конкатенировать) в один итоговый файл (например, `style.css`).

#### 2.1.2. Стратегия загрузки CSS на сайте (Оптимизация)

1.  **Стандартный подход: Единый файл (Рекомендуется по умолчанию)**

    Объединенный `style.css` подключается в `<head>` одним тегом `<link>`.
    -   **Плюсы:** Простота в реализации и поддержке.
    -   **Минусы:** Браузер должен скачать весь CSS-файл, прежде чем начать отображать страницу. Это может замедлить первую отрисовку.

2.  **Оптимизированный подход: Критический CSS (Для максимальной скорости)**

    Этот метод является лучшей современной практикой для ускорения загрузки.
    -   **Суть метода:**
        -   Определяется "критический CSS" — минимально необходимый набор стилей для корректного отображения верхней части страницы (того, что пользователь видит без прокрутки).
        -   Этот критический CSS встраивается (inline) прямо в HTML в тег `<style>` внутри `<head>`. Это устраняет блокирующий запрос к файлу стилей, и страница начинает отображаться почти мгновенно.
        -   Остальной CSS (все остальные стили) загружается асинхронно (не блокируя отображение страницы) и применяется после полной загрузки.
    -   **Плюсы:** Значительно ускоряет начальное отображение сайта.
    -   **Минусы:** Усложняет процесс сборки, так как требует автоматического определения критического CSS.

**Вывод и рекомендация:**

Для большинства проектов начинайте со стандартного подхода (единый `style.css`). Если производительность является ключевым требованием, применяйте оптимизацию с помощью Критического CSS.

### 2.2. Синтаксис и форматирование

-   **Отступы:** Используйте 4 пробела для вложенных правил.
-   **Пробелы:** Ставьте пробел после двоеточия (`color: #fff;`) и перед открывающей фигурной скобкой (`.selector {`).
-   **Новые строки:** Каждое объявление свойства должно быть на новой строке. Закрывающая фигурная скобка должна быть на новой строке.
-   **Точка с запятой:** Завершайте каждое объявление точкой с запятой, включая последнее.

✅ **Правильно:**

```css
.card {
    display: block;
    max-width: 300px;
    background-color: #f5f5f5;
}
```

### 2.3. Порядок свойств

Группируйте связанные свойства вместе. Рекомендуемый порядок основан на логике от "расположения" к "внутреннему содержанию":

-   **Позиционирование:** `position`, `top`, `right`, `bottom`, `left`, `z-index`.
-   **Блочная модель (Box Model):** `display`, `flex`, `grid`, `align-items`, `justify-content`, `width`, `height`, `margin`, `padding`, `border`, `box-sizing`.
-   **Типографика:** `font-family`, `font-size`, `font-weight`, `line-height`, `color`, `text-align`.
-   **Визуальное оформление:** `background-color`, `box-shadow`, `opacity`, `transform`, `transition`.
-   **Прочее:** `cursor`, `overflow`, `list-style`.

Пример:

```css
.button {
    /* Позиционирование */
    position: relative;
    top: 10px;

    /* Блочная модель */
    display: inline-block;
    padding: 10px 20px;
    border: 1px solid #ccc;

    /* Типографика */
    font-size: 16px;
    font-weight: 700;
    color: #333;

    /* Визуальное оформление */
    background-color: #eee;
    border-radius: 5px;
    transition: background-color 0.3s ease;

    /* Прочее */
    cursor: pointer;
}
```

### 2.4. Комментарии

Используйте комментарии для разделения больших логических блоков и объяснения неочевидного кода.

```css
/* ==========================================================================
   Header styles
   ========================================================================== */

.page-header {
    /* ... */
}

/* ==========================================================================
   Component: Card
   ========================================================================== */

.card {
    /* ... */
}
```

### 2.5. Единицы измерения

-   **Шрифты и отступы:** Используйте относительные единицы `rem` для `font-size`, `margin`, `padding`. Это улучшает доступность.
-   **Границы (borders):** Используйте `px` для тонких элементов, которым нужна точность.
-   **Макет (Layout):** Используйте `%` или `fr` (для Grid) для создания гибких сеток.

### 2.6. Специфичность и !important

-   **Низкая специфичность:** Старайтесь держать специфичность селекторов как можно ниже. Избегайте длинных цепочек селекторов и ID. Методология БЭМ помогает в этом.
-   **!important:** Использование `!important` запрещено, за исключением редких случаев переопределения стилей из сторонних библиотек.

## Часть 3: Использование CSS-препроцессоров (Sass/SCSS)

Для повышения эффективности и удобства поддержки CSS-кода в проектах обязательно используется препроцессор Sass с синтаксисом SCSS.

### 3.1. Основные правила

-   **Синтаксис:** Во всех проектах используется синтаксис `SCSS` из-за его совместимости с обычным CSS.
-   **Вложенность:** Глубина вложенности селекторов не должна превышать **3 уровня**. Это предотвращает избыточную специфичность и сохраняет CSS читаемым.

    ✅ **Правильно:**
    ```scss
    .card {
        // ...
        &__title {
            // ...
        }
    }
    ```

    ❌ **Неправильно (слишком глубокая вложенность):**
    ```scss
    .features {
        .features__list {
            .features__item {
                .card {
                    .card__title { // 5-й уровень вложенности
                        // ...
                    }
                }
            }
        }
    }
    ```
-   **Переменные:** Все повторяющиеся значения (цвета, размеры шрифтов, z-индексы, брейкпоинты) должны быть вынесены в переменные.
-   **Миксины:** Повторяющиеся группы свойств (например, стили для флекс-контейнера, медиа-запросы) должны быть вынесены в миксины.

### 3.2. Структура файлов (Архитектура 7-1)

Рекомендуется придерживаться архитектуры "7-1" для организации SCSS-файлов.

-   `abstracts/` (или `utils/`): Переменные, функции, миксины.
-   `base/`: Базовые стили, типографика, сбросы.
-   `components/`: Стили для отдельных компонентов (кнопки, карточки, формы).
-   `layout/`: Стили для основных частей макета (header, footer, grid, container).
-   `pages/`: Стили, специфичные для отдельных страниц.
-   `themes/`: Стили для различных тем оформления.
-   `vendors/`: Стили сторонних библиотек.
-   `main.scss`: Главный файл, который импортирует все остальные в правильном порядке.

## Часть 4: Подход к адаптивной верстке

### 4.1. Принцип Mobile-First

Все проекты разрабатываются по принципу **Mobile-First**. Это означает, что базовые стили пишутся для мобильных устройств, а затем расширяются для больших экранов с помощью медиа-запросов с `min-width`.

### 4.2. Брейкпоинты (Точки перелома)

В проекте используется единый набор брейкпоинтов, которые хранятся в файле с переменными (`abstracts/_variables.scss`).

```scss
// Пример
$breakpoint-md: 768px;
$breakpoint-lg: 1024px;
$breakpoint-xl: 1200px;
```

### 4.3. Расположение медиа-запросов

Медиа-запросы должны находиться **внутри** селектора компонента, к которому они относятся. Это улучшает инкапсуляцию и упрощает поддержку.

✅ **Правильно:**

```scss
.card {
    display: flex;
    flex-direction: column;

    @media (min-width: $breakpoint-md) {
        flex-direction: row;
    }

    &__title {
        font-size: 18px;

        @media (min-width: $breakpoint-md) {
            font-size: 22px;
        }
    }
}
```
